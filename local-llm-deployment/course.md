# 从容量评估到代理集成：本地大模型部署实战

## 课程定位

这是一门 12 小时、以交付和验收为中心的本地大模型部署课程。学生不需要拥有 256GB 内存：课堂实验使用适配学生电脑的 3B-8B 量化模型，教师用本次 DeepSeek V4 Flash 部署作为大型案例，讲清同一套方法如何扩展到 153GiB 权重、100 万上下文和常驻 API 服务。

课程不把“进程启动”当作部署完成。最终验收必须覆盖容量、下载、完整性、性能、服务、API、工具调用和客户端集成。

## 学习成果

完成课程后，学生能够：

1. 采集 CPU、GPU、统一内存或显存、磁盘和网络数据。
2. 根据实际权重文件、运行时开销、KV 缓存和系统余量判断模型是否可运行。
3. 解释模型参数量、MoE 激活参数、量化位宽、文件大小和推理速度之间的区别。
4. 为 Hugging Face 下载设计可验证的代理分流，而不是凭经验切换全局代理。
5. 选择合适的运行时，并配置 Metal、CUDA 或 CPU 后端。
6. 将模型部署为仅监听本机的常驻服务，配置日志、自动启动和故障恢复。
7. 验证 OpenAI 与 Anthropic Messages API，以及至少一次真实工具调用。
8. 将本地模型接入 Claude Code，并确认实际模型、上下文窗口和输出上限。
9. 用数据报告性能与限制，不夸大模型卡或客户端界面中的声明。

## 教学案例

本次实际部署的可公开课堂数据如下：

| 项目 | 实测值 |
|---|---:|
| 机器 | Apple M3 Ultra，256GB 统一内存，60 核 GPU |
| 模型 | Huihui DeepSeek V4 Flash abliterated，GGUF Q4_K |
| 权重文件 | 153GiB |
| 运行时 | ds4-server + Metal |
| 服务内存占用 | 约 153GiB RSS |
| 上下文配置 | 1,000,000 tokens |
| 服务最大输出配置 | 384,000 tokens |
| Claude Code 单次输出配置 | 65,536 tokens |
| 磁盘 KV 预算 | 64GiB，按需增长 |
| 长提示预填 | 约 360-400 tokens/s |
| 解码 | 约 34 tokens/s |
| API | OpenAI 风格接口 + Anthropic `/v1/messages` |
| 服务方式 | macOS LaunchAgent，RunAtLoad + KeepAlive |

注意：153GiB 是二进制单位显示值；下载站点显示的十进制大小会更大。课堂报告必须写清 GB 与 GiB，避免把单位差异误判为文件损坏。

## 模块 1：需求、边界与风险

### 讲授重点

先把“最强可运行”拆成约束，而不是直接给模型名称：

- 任务：编码、写作、研究、角色扮演或批量推理。
- 质量：模型能力、量化损失、工具调用可靠性。
- 交互：可接受的首 token 延迟和解码速度。
- 上下文：常用窗口、极限窗口和最大输出不是同一个参数。
- 隐私：是否必须完全离线，日志能否保存提示内容。
- 风险：模型许可、数据许可、校园规范和工具权限。

“无审查”或 abliterated 只描述模型行为倾向，不会取消法律、课程规范和工具安全边界。教学环境应默认监听 `127.0.0.1`，限制 Bash、文件写入和网络工具，不把服务暴露到公网。

### 课堂练习

让学生写一页部署需求单，至少包含：目标任务、上下文、并发、延迟、磁盘上限、是否离线、可接受量化和验收标准。

### 验收

学生能够解释为什么“能加载”不等于“能用于目标工作流”。

## 模块 2：硬件审计与容量预算

### 讲授重点

使用以下初步模型：

```text
内存需求 ~= 权重常驻 + 运行时开销 + 活跃 KV + 系统保留
磁盘需求 ~= 权重及下载余量 + 构建缓存 + 磁盘 KV 预算 + 日志余量
```

先采用实际制品大小，不要只用“参数量 × 位宽”估算。GGUF 还包含元数据、词表和不同张量的混合量化。

MoE 模型的“激活参数”主要影响每 token 计算量，不代表只需加载激活专家的权重。除非运行时明确支持专家按需加载或卸载，否则容量预算应按完整权重制品计算。

长上下文必须同时满足四个条件：

1. 模型和 RoPE 配置支持目标窗口。
2. 运行时允许目标窗口。
3. KV 缓存有足够内存或磁盘预算。
4. 客户端知道正确窗口，不会按较小的默认值提前压缩。

### 教师演示

本案例中 256GiB 统一内存减去约 153GiB 权重后，仍需给 macOS、Metal、运行时和活跃 KV 保留余量。64GiB 磁盘 KV 是容量上限，不等于启动时立刻占满 64GiB；应观察实际增长和性能。

### 学生实验

```bash
bash ~/.codex/skills/deploy-local-llm/scripts/audit_local_ai.sh

python3 ~/.codex/skills/deploy-local-llm/scripts/capacity_check.py \
  --ram-gib 32 \
  --weights-gib 5 \
  --kv-gib 4 \
  --disk-free-gib 100 \
  --kv-disk-gib 8
```

学生应替换为自己的数据，并记录所有假设。

### 验收

- 报告同时包含内存与磁盘预算。
- 保留系统余量，不以“恰好等于总内存”为可运行结论。
- 明确当前容量结论是估算还是实测。

## 模块 3：模型、量化与运行时选型

### 选型顺序

1. 明确任务能力和上下文需求。
2. 确认模型许可、来源和模型卡。
3. 确认运行时已经支持该架构，而非仅支持文件格式。
4. 比较实际量化制品大小与质量。
5. 用候选矩阵记录证据、风险和退路。

候选矩阵至少包含：模型、修订、量化、文件大小、目标运行时、上下文、工具调用、预计内存、预计速度和信息来源日期。

### 量化讨论

- BF16/FP16：质量高，容量要求最高。
- Q8：接近高精度，但文件仍大。
- Q5/Q6：质量与容量之间偏质量的选择。
- Q4：大型模型本地部署常见平衡点。
- Q2/Q3：只有在容量不足且经过任务评测后才采用。

量化名称不是跨项目的统一质量保证。同名量化也应通过目标任务评测。

### 运行时选择

- Apple Silicon：优先核实 MLX、llama.cpp Metal 或模型专用 Metal 运行时。
- NVIDIA：核实 CUDA、显存分层、量化内核和多 GPU 支持。
- CPU：关注内存带宽、线程拓扑和首 token 延迟。
- 新模型架构：优先使用已经声明并测试该架构的运行时，避免强行加载。

### 验收

每组提交一个主方案和一个降级方案，并说明为什么不选择更大的参数量或更低的量化。

## 模块 4：下载、网络分流与完整性

### 关键经验

下载瓶颈必须按域名和实际文件链路定位。模型页面、鉴权、重定向与大权重 CDN 可能走不同域名。

本次网络实测得到的规则是：

```yaml
- DOMAIN-SUFFIX,xethub.hf.co,DIRECT
- DOMAIN-SUFFIX,huggingface.co,AI_PROXY
- DOMAIN-SUFFIX,hf.co,AI_PROXY
```

更具体的 `xethub.hf.co` 必须排在 `hf.co` 泛规则之前。这个结论只能作为方法案例；换网络、运营商或时间后必须重新测速，不能永久照抄。

### 实验流程

1. 测试模型页面和鉴权链路。
2. 获取真实权重重定向后的 CDN 域名。
3. 分别测试直连与代理的吞吐、稳定性和失败率。
4. 只修改必要域名的路由。
5. 使用支持断点续传和并行分块的下载器。
6. 对下载结果执行大小、校验值和格式检查。

不要把 Hugging Face 全部设为直连：在本案例网络中，`huggingface.co` 直连会被重置，而 Xet CDN 直连显著更快。

### 验收

学生提交测速证据、最终路由规则、规则优先级解释和完整性校验结果。

## 模块 5：首次运行与性能调优

### 最小化启动

先使用保守上下文和短输出完成冒烟测试，再逐步放大：

1. 加载权重并生成固定短句。
2. 开启 GPU 或统一内存后端。
3. 测量冷启动、预填和解码。
4. 增加上下文与 KV 预算。
5. 开启权重预热和持久 KV。
6. 对目标任务重新测量。

每次只改变一个变量，否则无法解释性能变化。

### 必测指标

- 冷启动时间和权重预热时间。
- 首 token 延迟。
- 短提示与长提示的预填 tokens/s。
- 解码 tokens/s。
- 峰值内存、磁盘 KV 增长和磁盘吞吐。
- 10 次连续请求的稳定性。

### 本案例解释

34 tokens/s 是短交互中观察到的解码速度，不应直接外推到 100 万 token 满上下文。上下文越长，KV 访问、缓存命中和首 token 延迟可能改变。必须按 1K、8K、32K、128K 等梯度测试。

## 模块 6：常驻服务与运维

### 服务工程原则

- 默认绑定 `127.0.0.1`，确需局域网访问时再增加鉴权和防火墙。
- 使用绝对路径，避免守护进程缺少交互式 shell 的 PATH。
- 分离启动脚本、服务定义、日志和模型文件。
- 配置开机启动、异常重启、节流和优雅停止。
- 启动前检查模型文件存在，失败时给出明确错误。

本案例使用 macOS LaunchAgent：`RunAtLoad=true`、`KeepAlive=true`、固定工作目录、独立 stdout/stderr。Linux 可使用 systemd 实现相同职责。

### 分层验收

```text
进程存在 -> 端口监听 -> /v1/models -> 文本生成 -> 流式生成
          -> Anthropic Messages -> 工具调用 -> 真实客户端
```

只检查 PID 会漏掉死锁、错误端口、错误模型和协议不兼容。

## 模块 7：API 与 Claude Code 集成

### Anthropic Messages 冒烟测试

```bash
bash ~/.codex/skills/deploy-local-llm/scripts/probe_anthropic_backend.sh \
  --base-url http://127.0.0.1:8000 \
  --model deepseek-v4-flash
```

Claude Code 会在 `ANTHROPIC_BASE_URL` 后追加 `/v1/messages`。如果本地服务原生提供 `/v1/messages`，根地址应为 `http://127.0.0.1:8000`，不能机械照搬云端供应商的 `/anthropic` 前缀。

### Claude Code 用户配置示例

```json
{
  "model": "deepseek-v4-pro[1m]",
  "env": {
    "ANTHROPIC_BASE_URL": "http://127.0.0.1:8000",
    "ANTHROPIC_AUTH_TOKEN": "local-only",
    "ANTHROPIC_MODEL": "deepseek-v4-pro[1m]",
    "ANTHROPIC_DEFAULT_OPUS_MODEL": "deepseek-v4-pro[1m]",
    "ANTHROPIC_DEFAULT_SONNET_MODEL": "deepseek-v4-pro[1m]",
    "ANTHROPIC_DEFAULT_HAIKU_MODEL": "deepseek-v4-flash",
    "CLAUDE_CODE_SUBAGENT_MODEL": "deepseek-v4-flash",
    "CLAUDE_CODE_EFFORT_LEVEL": "max",
    "CLAUDE_CODE_MAX_CONTEXT_TOKENS": "1000000",
    "CLAUDE_CODE_MAX_OUTPUT_TOKENS": "65536",
    "API_TIMEOUT_MS": "1800000",
    "CLAUDE_STREAM_IDLE_TIMEOUT_MS": "1800000",
    "ENABLE_TOOL_SEARCH": "false"
  }
}
```

配置值必须按实际后端调整。不要把 `384000` 服务输出能力直接设成 Claude Code 默认输出：过大的保留输出会减少压缩前的有效上下文，64K 对代码代理是更稳妥的起点。

### 端到端验收

```bash
claude -p --tools Bash --allowedTools 'Bash(pwd)' \
  'Use Bash exactly once to run pwd, then return only the path.'
```

验收日志应证明：请求到达本地端口、模型完成 `tool_use`、工具结果返回模型、第二轮生成成功。

## 模块 8：故障演练与结课考核

### 故障演练

教师随机注入一个故障：

- 错误模型路径。
- 端口已占用。
- 上下文配置超过实际容量。
- `ANTHROPIC_BASE_URL` 多写了一层路径。
- Claude Code 仍选择云端默认模型。
- Hugging Face 规则顺序错误。
- 下载文件大小正确但校验失败。
- 服务进程存在但 `/v1/messages` 不兼容。

学生必须按证据链排查，不能直接重装全部组件。

### 评分标准

| 项目 | 权重 | 达标要求 |
|---|---:|---|
| 容量评估 | 20% | 数据真实、假设明确、保留余量 |
| 选型论证 | 15% | 有主方案、降级方案和当前来源 |
| 下载与完整性 | 15% | 有测速、分流和校验证据 |
| 服务工程 | 15% | 可启动、停止、重启、自动恢复 |
| API 与工具调用 | 20% | Messages API 与真实工具闭环成功 |
| 性能报告 | 10% | 区分预填、解码、冷启动和上下文 |
| 安全与表达 | 5% | 不泄露凭据，不暴露公网，结论可复现 |

### 结课标准

学生现场执行一次全新终端会话：检查服务、调用 API、运行 Claude Code 工具任务、停止并恢复服务。四步均成功才算部署完成。

## 故障排查速查表

| 症状 | 先检查 | 常见原因 |
|---|---|---|
| 模型无法加载 | 文件大小、格式、运行时架构支持 | 下载不完整、量化不支持 |
| 系统严重换页 | 权重 RSS、KV、系统余量 | 容量只按文件大小计算 |
| 首 token 很慢 | 冷启动、预填速度、上下文长度 | 未预热、提示过长、磁盘 KV |
| 页面能打开但权重极慢 | 重定向后的 CDN 路由 | 所有 HF 域名走了同一代理 |
| API 404 | 完整请求路径 | Base URL 多一层或少一层 |
| Claude Code 要求登录 | AUTH_TOKEN 与配置来源 | 配置未加载或被其他设置覆盖 |
| Claude Code 提前压缩 | 客户端识别的上下文 | 未设置自定义模型窗口 |
| 只能聊天不能编码 | 工具调用测试 | 后端未兼容 Anthropic tool_use |
| PID 存在但请求失败 | 端口和健康接口 | 服务卡死、错误监听地址 |

## 教师总结

这次部署最值得教给学生的不是某一组命令，而是五条工作方法：

1. 用实际制品和实测日志代替参数量想象。
2. 把上下文能力拆成模型、运行时、KV 和客户端四层。
3. 把网络按真实下载链路拆分，不做粗暴的全局直连或全局代理。
4. 把服务验收推进到真实工具调用，不停在 PID 或聊天回复。
5. 把配置做成可启动、可观察、可回滚的系统，而不是一次性终端会话。
